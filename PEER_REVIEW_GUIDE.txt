============================================================
  PEER REVIEW GUIDE: Smart Path Finder (End-to-End Approach)
============================================================

This document provides a technical overview of our implementation for the
Smart Path Finder project to facilitate peer review.

1. CORE SYSTEM ARCHITECTURE
---------------------------
The project is built using a modular, "no-library" approach. We implemented 
all required data structures (Graph, MinHeap) from scratch to ensure maximum 
control over performance and memory.

- graph.py: Adjacency list representation (O(V+E) space).
- heap.py:  Custom binary min-heap with O(log V) decrease_key support.
- algorithms.py: Dijkstra (Distance/Time) and Bidirectional Dijkstra.
- cache.py: Path caching, hub identification, and precomputation.
- generator.py: Synthetic graph generator with 24h traffic profiles.

2. DATA STRUCTURES & LOGIC
--------------------------
* Adjacency List: Each node maps to a list of neighbors. We chose this over 
  an adjacency matrix because road networks are sparse. Each edge carries a 
  24-element list of travel times (one per hour).

* Custom MinHeap: Standard Python heaps don't support `decrease_key`. Our 
  implementation uses a dictionary to track node positions in the heap array, 
  ensuring Dijkstra runs in O((V+E) log V) instead of O(V^2).

3. ROUTING ALGORITHMS
---------------------
* Dijkstra (Distance): Standard implementation for static shortest paths.
* Dijkstra (Time): Dynamic implementation. When relaxing an edge, it 
  calculates the "Current Hour" based on (Departure Hour + Time Elapsed) to 
  pick the correct weight from the 24h profile.
* Bidirectional Dijkstra: Simultaneously searches from source and 
  destination. In our benchmarks, this reduces node exploration by ~35% 
  and provides a ~1.5x speedup for distance queries.

4. PERFORMANCE OPTIMIZATIONS (PRECOMPUTATION & CACHING)
------------------------------------------------------
To handle a 4,900-node graph efficiently:
* Hub Identification: We identify "Hubs" based on node degree (number of 
  connections). These represent major intersections.
* Precomputation: At startup, we pre-calculate paths between all hub pairs 
  for peak and off-peak hours. This populates ~6,960 entries.
* Result Caching: All unique queries (including those with avoid_nodes/edges) 
  are stored in an LRU-style cache to ensure repeated searches return in O(1).

5. SYNTHETIC DATA GENERATION
----------------------------
The map is a 70x70 grid (4,900 nodes). We add:
- Local roads (grid edges).
- Random diagonal shortcuts.
- Long-range Highway Edges: Euclidean-based shortcuts that allow crossing 
  the map in far fewer hops (demonstrated in Evaluation Scenario 4).
- Traffic Profiles: Each edge is slow at hours 7-9 and 17-19.

6. EVALUATION STRATEGY
----------------------
We validated the system using 7 distinct scenarios in `evaluation.py`:
- Random Query Baseline (O((V+E) log V) verification).
- Cache Effectiveness (measured at 18,000x speedup).
- Avoidance Constraints (testing query flexibility).
- Departure Hour Impact (demonstrating traffic awareness).
- Scaling Analysis (testing performance from 400 to 10,000 nodes).

7. KEY DESIGN DECISIONS
-----------------------
- Why no Bidirectional Time search? Time weights are asymmetric and 
  direction-dependent; a backward search wouldn't know the "Arrival Hour" 
  in advance without a forward pass.
- Why degree-based hubs? It's a computationally cheap way to find 
  bottlenecks without the O(V^3) cost of Betweenness Centrality.
